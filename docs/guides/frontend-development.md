# Frontend Development Guide

GridFleet's frontend acts as a high-density operator workspace. It runs on **React + TypeScript + Vite + TailwindCSS**.

## App Architecture

```text
frontend/src/
├── api/          # Strongly typed Axios clients matching backend routes
├── components/   # Domain/feature components
│   └── ui/       # Shared UI primitives (DataTable, Badge, Button, FilterBar)
├── context/      # Global React providers (auth, theme)
├── hooks/        # react-query data hooks with defined polling intervals
├── lib/          # Framework-agnostic helpers (state derivation, telemetry, errors)
├── pages/        # Routable top-level views (Dashboard, Devices, Hosts)
├── types/        # Stable frontend aliases derived from generated OpenAPI schemas
└── utils/        # Small formatting/value helpers
```

- **`src/api/`**: Strongly typed Axios clients. All external communication routes through here.
- **`src/components/`**: Houses domain/feature components, with reusable UI primitives under `src/components/ui/`. Ensure you reuse existing `components/ui/` primitives (like `Badge`, `DataTable`, `FilterBar`) before building new ones.
- **`src/hooks/`**: Data fetching and mutation hooks using `react-query`. Freshness is SSE-primary: the `/api/events` stream (mounted once in `Layout`) invalidates the relevant query caches on transitions. Polling is a safety-net backstop via `sseAdaptivePolling` (`src/hooks/polling.ts`) — while the stream is connected each hook relaxes to a 60s poll, and only falls back to its 5–30s base interval when SSE is disconnected.
- **`src/pages/`**: Routable views built by combining layout frames with atomic UI components.
- **`src/api/openapi.ts`**: Generated by `openapi-typescript` from the backend app. This file is committed and ignored by ESLint because it is generated.
- **`src/types/`**: Stable public type names for the frontend. DTOs should derive from `components["schemas"]` in `src/api/openapi.ts`; keep handwritten types only for frontend-only query params, sort keys, and narrow refinements around intentionally dynamic JSON payloads.

## API Type Generation

Public backend responses that the frontend consumes should use named Pydantic `response_model`s. The frontend should derive DTOs from OpenAPI `components["schemas"]`, not from route-specific `paths[...]` response shapes. Dynamic backend subfields such as grid runtime payloads, settings validation metadata, or driver-pack manifest maps can remain loose in the backend schema and be narrowed in `src/types/` where the UI needs structured access.

Regenerate types after backend API schema changes:

```bash
cd frontend
npm run types:generate
npm run types:check
```

`types:generate` writes a temporary `frontend/openapi.json` and regenerates `src/api/openapi.ts`. `types:check` also fails if the committed generated file is out of date.

## Design Patterns & Standards

During Phase 109 and subsequent updates, the frontend styling was unified under a standard design token set. If developing new components, strictly adhere to these practices:

### 1. Spacing and Density
- We utilize standard Tailwind spacing scales (`p-4`, `p-6`). Never hardcode pixel values when a Tailwind class is available.
- Keep the density appropriate for an "Operator Console": compact tables, clear hierarchical alerts, and structured data layouts.

### 2. Form Controls
- Use native HTML elements standardly wrapped. E.g., for dates use the `<DateInput>` primitive from `components/ui`; a raw `<input type="date">` is blocked by an ESLint `no-restricted-syntax` rule (exempt only inside `components/ui/`).
- Prefer unified payloads for complex search filters and entity creation.

### 3. Error states & Fallbacks
- **`FetchError`**: Always wrap queries that may fail with the standard `<FetchError message="..." onRetry={refetch} />` component. The retry callback prop is `onRetry` (required); `message` is optional and defaults to a generic message.
- Loading states should ideally prevent jarring layout shifts (e.g., use block skeleton loaders if fetching large data segments).

## Development Workflow

To build and run locally:

```bash
cd frontend
npm ci
npm run dev
```

### Static Analysis
Ensure code passes existing linting rules before committing:
```bash
npm run lint         # Standard ESLint
npx tsc --noEmit     # TypeScript type-check
npm run types:check  # OpenAPI generation drift check
```

### E2E Tests
Browser-scoped tests run via Playwright. `npm run test:e2e` runs `test:e2e:mocked` then `test:e2e:live`:

- `test:e2e:mocked` runs against a mocked backend and needs only the frontend.
- `test:e2e:live` needs a real backend (which itself requires Postgres).

Playwright's `webServer` config launches the required servers automatically — the live config starts the backend (`uvicorn` on `:8000`) and the frontend dev server — reusing already-running ones only outside CI, so you do not need to start them by hand.
```bash
npm run test:e2e          # mocked, then live
npm run test:e2e:mocked   # mocked backend, frontend only
npm run test:e2e:live     # real backend (+ Postgres)
```
